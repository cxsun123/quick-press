> 日期：2026-07-20

## 1. 概述

本文档覆盖 4 大新需求：编辑界面优化、AI 摘要、Agent 集成 (MCP Server)、文章可见度。

### 1.1 与 v0.2 的关系

v0.2 已完善博客 CMS 基础功能（文章/页面/分类/标签/评论/主题/用户）。v0.3 在此之上增强编辑体验、引入 AI 能力和 API 对接。

## 2. 编辑界面优化

### 2.1 左侧菜单可最小化

**当前状态**：AdminLayout 侧边栏固定 `w-56`（224px），无可折叠功能，无响应式适配。

**需求**：

| 项目 | 规格 |
|------|------|
| 折叠按钮 | 侧边栏顶端/底部新增图标按钮 (`PanelLeftClose` / `PanelLeftOpen`) |
| 展开宽度 | `w-56` (224px)，显示图标 + 文字 |
| 折叠宽度 | `w-14` (56px)，仅显示图标 |
| 过渡动画 | `transition-all duration-200 ease-in-out` |
| 图标模式 Tooltip | 鼠标悬停折叠态菜单项时，在右侧弹出 tooltip 显示菜单名 |
| 折叠状态持久化 | `localStorage` 存储 `sidebar_collapsed` 状态，刷新后保留 |
| 移动端 (<768px) | 侧边栏变为覆盖式抽屉，背景半透明遮罩 (overlay)，点击遮罩关闭 |

**交互规则**：
- 点击折叠按钮 → 切换展开/折叠状态
- 折叠状态下 hover 侧边栏边缘可选临时展开（产品决策：不做过度设计，仅按钮切换）

### 2.2 文章编辑界面布局优化

**当前状态**：PostEditor 在 AdminLayout 中渲染，左侧 224px 侧边栏 + 右侧内容区，标题/编辑/标签/分类堆叠排列。

**需求**：

#### 2.2.1 左侧菜单自动折叠

进入以下路径时，侧边栏自动折叠（最小化为图标模式）：
- `/admin/posts/new`
- `/admin/posts/[id]/edit`
- `/admin/pages/new`
- `/admin/pages/[id]/edit`

离开上述路径时，恢复用户偏好状态（localStorage 中保存的状态）。

自动折叠仅触发展开→折叠，不阻止用户手动展开。

## 3. AI 摘要

### 3.1 功能描述

文章除了 title、content，加入 summary (摘要) 和 keywords (关键字) 属性。用户可手动输入摘要，也可点击"提取摘要"按钮，通过 AI 自动生成摘要和关键字列表。

### 3.2 数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| `summary` | TEXT, nullable | 文章摘要 (50 字以内，AI 生成或手工填写) |
| `keywords` | TEXT[] (PostgreSQL 数组), nullable | 3-5 个关键字 |

### 3.3 AI Provider 配置

**全局配置**，管理员在 `/admin/settings` 页面配置：

| 配置项 | Key | 说明 |
|--------|-----|------|
| Provider URL | `ai_provider_url` | OpenAI 兼容 API 地址 (如 `https://api.openai.com/v1/chat/completions`) |
| API Key | `ai_api_key` | 仅服务端可见，前端输入框以 password 类型呈现 |
| Model | `ai_model` | 模型名称，默认 `gpt-4o-mini` |

存储于 `site_config` 表，`ai_api_key` 不暴露给客户端（设置页读取时脱敏展示）。

### 3.4 摘要提取交互

1. 用户在文章编辑器右侧面板的"摘要"区块中点击"提取摘要"按钮
2. 前端调用 Server Action `extractSummary(content)`
3. 服务端：
   a. 从 `site_config` 读取 AI provider 配置
   b. 使用预设 prompt 模板构造请求
   c. 调用 OpenAI 兼容 API
   d. 解析返回的 JSON `{ summary, keywords }`
   e. 返回给前端
4. 自动填入 summary textarea 和 keywords 区域
5. 用户可手动修改 AI 生成的结果
6. 保存文章时，summary 和 keywords 随 title/content 一并提交

**按钮状态**：
- 空闲态：显示"提取摘要"
- Loading：按钮 disabled + 显示"提取中..."
- 成功：summary/keywords 自动填入，按钮短暂变绿 "已提取"
- 失败：toast 提示错误信息

### 3.5 Prompt 模板

```
Task: Extract the summary and keywords from the article below.

Input:
{
  "article": "${content}"
}

Output Requirements:
- Return ONLY a JSON object containing summary and keywords.
- Summary: A single sentence of fewer than 50 characters.
- Keywords: 3–5 keywords.
- Output Language: Chinese.
- Content: The response should begin with { and end with }.
- Do not include any introductory text, markdown explanations, or notes.
- No Markdown: Do not wrap the JSON in markdown formatting.
- Strict Output Schema:
    {
      "summary": "本文是关于数组解构、合并展开及reduce的统计",
      "keywords": ["Name 1", "Name 2"]
    }
```

### 3.6 安全与限制

- AI API Key 仅存服务端，前端无法读取
- 请求频率限制：每用户每小时最多 20 次（通过内存计数器，远期可改 Redis）
- 文章 content 超过 8000 字符时，截取前 8000 字符发送（避免 token 超限）
- API 调用超时 30 秒

## 4. Agent 集成 (MCP Server)

### 4.1 目标

实现 MCP (Model Context Protocol) Server 端点，用户通过 Claude Code、Codex、Cursor 等 MCP 客户端连接博客系统，以自然语言指令管理文章。

### 4.2 实现方式

- **Next.js API Route**：`src/app/api/mcp/route.ts`
- **协议**：MCP Streamable HTTP (2025-03-26 规范)
- **传输**：HTTP POST + SSE (Server-Sent Events) 流式响应
- **端点**：`POST /api/mcp`

### 4.3 支持的 MCP Tools

| 工具名 | 功能 | 输入参数 | 输出 |
|--------|------|---------|------|
| `create_draft` | 创建新文章草稿 | `title`(必填), `content`?, `tags`?, `categories`?, `visibility`? | `{ postId, slug, url }` |
| `publish_post` | 发布或更新文章 | `postId`(必填), `title`?, `content`?, `tags`?, `categories`?, `visibility`? | `{ postId, slug, url }` |
| `list_posts` | 列出文章 | `status`?, `visibility`?, `limit`?, `offset`? | `{ posts[], total }` |
| `get_post` | 获取文章详情 | `postId`(必填) | `Post` 完整对象 |
| `delete_post` | 删除文章 | `postId`(必填) | `{ success: true }` |
| `search_posts` | 搜索文章 | `query`(必填), `limit`? | `{ posts[] }` |
| `get_stats` | 获取博客统计 | 无 | `{ totalPosts, publishedPosts, scheduledPosts, totalComments }` |

### 4.4 认证

| 项目 | 规格 |
|------|------|
| 认证方式 | `Authorization: Bearer <mcp_api_key>` |
| Key 存储 | `site_config` 表 `mcp_api_key` |
| Key 生成 | `/admin/settings` 中由管理员生成 (128-bit 随机字符串) |
| 权限级别 | mcp_api_key 对应 Editor 级别权限 |
| Key 管理 | 管理员可查看 / 重新生成 / 删除 Key |

### 4.5 客户端配置示例

Claude Code `.mcp.json`：
```json
{
  "mcpServers": {
    "i_blog": {
      "url": "https://blog.example.com/api/mcp",
      "headers": {
        "Authorization": "Bearer sk-mcp-xxxxxxxxxxxx"
      }
    }
  }
}
```

### 4.6 MCP 协议交互流程

```
Client → POST /api/mcp
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}

Server → SSE
data: {"jsonrpc":"2.0","id":1,"result":{"tools":[...]}}

Client → POST /api/mcp
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": { "name": "create_draft", "arguments": { "title": "Hello" } }
}

Server → SSE
data: {"jsonrpc":"2.0","id":2,"result":{"content":[{"type":"text","text":"..."}]}}
```

## 5. 文章可见度

### 5.1 可见度级别

| 级别 | 标识 | 说明 | 前端行为 |
|------|------|------|---------|
| 公开 | `public` | 所有人可见 | 出现在首页/归档/搜索/RSS |
| 私密 | `private` | 仅作者和管理员可见 | 不在公开列表显示；访问返回 404 |
| 密码保护 | `password` | 输入正确密码可见 | 不在公开列表；访问时显示密码输入页面 |

### 5.2 数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| `visibility` | VARCHAR(20), DEFAULT 'public' | CHECK (visibility IN ('public', 'private', 'password')) |
| `password_hash` | VARCHAR(255), nullable | bcrypt 哈希后的密码（仅 password 模式时使用） |

### 5.3 可见度设置交互

#### 文章编辑页
- 右侧面板 → 可见度区块
- 三选一控件（Radio 按钮组或 Select 下拉）
- 默认选中"公开"
- 选择"密码保护"时，下方弹出密码输入框（明文输入，服务端 bcrypt 哈希存储）

#### 文章列表页
- 每篇文章行显示可见度标签（公开:绿色 / 私密:灰色 / 密码保护:橙色）
- 新增可见度筛选下拉框
- 支持批量操作：复选框多选 → 底部 BatchActionBar → "批量设为公开" / "批量设为私密"
- 密码保护文章不可批量设置（密码唯一，需单篇设置）

### 5.4 密码保护文章访问流程

```
用户访问 /blog/[slug]
  → SSR 阶段：select 仅返回 visibility = 'public' 的 published 文章
  → post 为 null：
    → 服务端用 admin client 重新查询完整信息
    → 如果是 visibility = 'password' 的文章：
      → 返回 PasswordGate 页面（密码输入框）
      → 用户输入密码 → POST /api/verify-password { postId, password }
        → 服务端 bcrypt.compare(password, post.password_hash)
        → 匹配成功 → 设置 session cookie (post_read_token)
        → 返回 200 + 文章内容
        → 前端重新渲染文章内容
      → 密码错误 → 返回 401 + 错误提示
    → 如果是 visibility = 'private'：
      → 返回 404 页面
```

### 5.6 分享链接

仅密码保护文章可生成分享链接，拥有链接的人无需输入密码即可直接查看全文。

| 项目 | 规格 |
|------|------|
| 适用文章 | 仅 visibility = 'password' 的文章 |
| Token 生成 | 首次设置密码时自动用 `crypto.randomUUID()` 生成，存入 `posts.share_token` |
| Token 策略 | 固定不变，除非作者手动"重新生成" |
| 有效期 | 永久 |
| 链接格式 | `${window.location.origin}/blog/{slug}?token={share_token}`（前端直接构造，无需 `site_url` 配置） |

**访问流程**：
```
用户访问 /blog/[slug]?token=xxx
  → 服务端查询 posts.share_token = xxx
  → 匹配成功 → 跳过密码验证，直接渲染文章内容
  → 匹配失败/无 token → 正常渲染 PasswordGate
```

**管理 UI**（VisibilityPanel，密码保护模式选中后）：
```
🔗 分享链接: https://blog.example.com/blog/slug?token=xxx  [📋 复制]  [🔄 重新生成]
```

- 复制按钮：`navigator.clipboard.writeText(shareUrl)` + toast 提示"已复制"
- 重新生成：确认后调用 Server Action 重新生成 token，旧链接立即失效

**安全**：`share_token` 为 128-bit 随机 UUID，不可预测，不可枚举。

### 5.5 RLS 策略调整

```sql
-- 修改 posts 公开读取策略
CREATE POLICY "Posts public read" ON posts FOR SELECT USING (
  status = 'published'
  AND (visibility = 'public' OR visibility IS NULL OR visibility = '')
);
```

## 6. 非功能性需求

### 6.1 性能
- 侧边栏折叠/展开动画帧率 ≥ 30fps
- AI 摘要提取超时 30 秒，超时返回友好错误
- MCP 端点请求处理延迟 ≤ 5s (不含 AI)

### 6.2 安全
- `ai_api_key` 和 `mcp_api_key` 仅服务端可见，不在前端渲染
- `password_hash` 使用 bcrypt (cost=12)
- MCP 端点 rate limiting：每 key 每分钟 30 次

### 6.3 兼容性
- 现有的文章/页面/标签/分类功能不受影响
- 旧文章默认 `visibility = 'public'`，`summary = null`，`keywords = null`
- 管理员列表页向后兼容（新列 nullable）

### 6.4 响应式
- 侧边栏移动端自适应
- 编辑器三栏布局在 <1024px 时右侧面板变为底部弹出或可折叠
- 密码验证页面对移动端友好
